## 一、 文本的序列化：分词（Tokenization）

> **💡 原文回放**：ai是没法认识文字的...第一步是将文字转换为数字。ai处理文字时，不是一个个字母进行识别的，也不是单纯的一个单词，而是token。token是根据分词器（chatgpt使用的是bpe）产出的ai可以处理的最小单位。
> **🔍 补充与扩展**：这里提到了一个核心概念：**Token（词元）**。为什么不直接用字母（Char）或者单词（Word）？

在计算机科学中，我们需要将非结构化的字符串转化为结构化的整数数组（ID序列）。

1. **为什么不用字母（Char）？**
   - 字母缺乏语义。单个字母 `a` 或 `p` 没有独立含义，且如果把文章拆成字母，数组长度会暴增，导致 AI 处理的上下文窗口（Context Window）迅速被耗尽。
2. **为什么不用单词（Word）？**
   - 英文单词有无数的变体（如 `look, looking, looked`），中文更是没有空格分隔。如果以单词为单位，词表（Vocabulary）会无限膨胀，遇到没见过的生僻词就会产生 **OOV（Out Of Vocabulary，未登录词）** 错误。

**工程妥协的最优解：BPE（Byte Pair Encoding，字节对编码）**
ChatGPT 等大模型普遍使用 BPE 算法。它是一种**子词（Sub-word）**分词法，结合了字母和单词的优点：它将高频词作为一个整体，将低频词拆分成常见的词根或音节。

*(工程师类比：类似于数据压缩中的霍夫曼编码或字典压缩算法)*
```text
输入文本: "unhappiness"
如果按单词: ["unhappiness"] (生僻词，可能不在词表里)
如果按 BPE: ["un", "happi", "ness"] (被拆分为三个常见 token)
转化为数字 ID: [123, 4567, 89]
```

### 深入解析：BPE 分词器是如何被“训练”出来的？

作为开发者，我们可以把 BPE 理解为一个**贪心算法（Greedy Algorithm）**，它的目标是找出文本中最常出现的字节对（相邻的字符组合），并将它们合并成一个新的 Token。

**核心训练流程（算法视角解析）**：
1. **初始化词表**：把训练语料中所有的单词拆成单个字母（或字节），并在单词末尾加上特殊停止符（如 `</w>`）。此时词表里只有基础字母。
2. **频率统计**：扫描整个语料库，统计所有**相邻两个符号**的出现频率。
3. **合并最高频对**：找到频率最高的那一对，将它们合并成一个新符号，并加入词表。
4. **迭代**：重复步骤 2 和 3，直到词表达到预设的大小限制（比如 OpenAI 的词表大小通常是 10万 级别）。

**代码级运行实例演示**：
假设我们的语料库里只有这几个词和它们的出现次数：
- `low` (5次), `lower` (2次), `newest` (6次), `widest` (3次)

*初始状态*：拆分成字母 `l, o, w, e, r, n, s, t, i, d`。
*第 1 轮合并*：算法发现 `e` 和 `s` 相邻出现了 9 次（newest 6次 + widest 3次），频率最高。于是合并出新的 Token：`es`。
*第 2 轮合并*：接着发现 `es` 和 `t` 相邻出现了 9 次。合并出新的 Token：`est`。
*第 3 轮合并*：`l` 和 `o` 出现了 7 次（low 5次 + lower 2次）。合并出新的 Token：`lo`。

最终，`est` 这个常见的英文后缀就被提取成了一个独立的 Token。当遇到完全没见过的新词 `highest` 时，哪怕词库里没有这个完整单词，分词器也能将其拆分为 `high` 和 `est` 两个已知的 Token，完美避免了程序报错崩溃（OOV 问题）。

## 二、 赋予数字以意义：词向量（Word Embeddings）

将文字变成整数 ID 后，AI 依然无法理解含义。数字 `123` 和数字 `124` 在数学上大小相近，但它们代表的 Token 含义可能天差地别。

> **💡 原文回放**：ai怎么学习到含义？词向量（根据训练上下文经常出现的单词相邻的关系而产生的数据矩阵）
> **🔍 科普补充**：这在语言学上叫作**分布假说 (Distributional Hypothesis)** —— *“一个词的含义由它的上下文决定”*（You shall know a word by the company it keeps. —— J.R. Firth, 1957）。

在代码世界里，我们往往用 `Enum` 或 `Map` 来定义状态。而在 AI 世界，**词向量（Embedding）** 是一个高维的浮点数数组（通常有 300 到数千个维度）。
模型通过阅读海量文章，如果发现“猫”和“狗”总是出现在相似的上下文中（比如“___在吃猫粮/狗粮”），它就会在多维空间中把这两个词的坐标拉近。

### 深入解析：词向量是如何被“训练”出来的？

你可能会好奇：那些包含 300 个维度的浮点数矩阵，到底是怎么算出来的？是谁告诉 AI 哪个词和哪个词相似？
答案是：**不需要人工打标签，利用上下文做“自监督学习（Self-supervised Learning）”。**

这里以经典的 **Word2Vec (Skip-gram 模型)** 为例，它的核心思想是：**用中心词去预测周围的词**。

**训练过程（神经网络视角的工程实现）**：
1. **滑动窗口（Sliding Window）构建数据集**：
   假设我们有一句话：`"The quick brown fox jumps over the lazy dog"`。
   我们设定一个大小为 2 的窗口。当中心词是 `fox` 时，它的上下文是 `["quick", "brown", "jumps", "over"]`。
   此时我们就构造出了 4 条训练数据：`(fox, quick)`, `(fox, brown)`, `(fox, jumps)`, `(fox, over)`。
2. **独热编码（One-Hot Encoding）输入**：
   最开始，所有词都被表示为一个超长的 0/1 数组（长度为词表大小，假设有 10000 个词）。`fox` 可能是 `[0, 0, 1, 0, ... 0]`。
3. **构建极简的单隐层神经网络**：
   - **输入层**：10000 维（One-Hot 向量）
   - **隐藏层**：300 维（**注意：这里没有激活函数，纯线性映射！**）
   - **输出层**：10000 维（经过 Softmax，输出预测每一个词的概率）
4. **反向传播与“买椟还珠”**：
   我们把 `fox` 的 One-Hot 向量输入网络，强迫网络去预测 `brown` 或 `jumps`。一开始网络是瞎猜的，预测错误后，系统通过**损失函数**和**反向传播**来调整网络中的权重参数。

**最巧妙的工程设计来了**：训练结束后，我们**直接丢弃输出层！** 我们根本不在乎这个网络最终预测得准不准。
我们真正想要的是**输入层到隐藏层之间的权重矩阵（Weight Matrix）**。这个矩阵的大小刚好是 `10000 × 300`。
这个矩阵，就是我们梦寐以求的**词向量表（Embedding Lookup Table）**！矩阵的第 N 行，就是第 N 个词的 300 维词向量。在不断“做预测选择题”的过程中，经常搭配出现的词，它们在隐藏层中的参数会被更新得越来越相近。

## 三、 隐式特征空间：King - Man + Woman = Queen

> **💡 原文回放**：依据word2vec模型的故事，他们每个token有300维度，每个维度代表一个特征，这些维度和特征都是机器学习得到的，人类无法直接理解。研究员发现 国王-男人+女人=皇后...

这是 NLP 领域最著名的数学实验之一！

**程序员视角解析：**
假设这 300 个维度（`[float, float, ...]`）中，某几个维度隐式代表了“性别（Gender）”，某几个维度隐式代表了“权力地位（Royalty）”。
虽然这 300 个特征是不可解释的（黑盒），但它们构成了具有线性代数性质的**连续向量空间**。

```python
# 伪代码：词向量的代数运算
vector_king   = [0.9, 0.1, 0.8, ...] # [地位高, 女性特征低, 男性特征高...]
vector_man    = [0.1, 0.1, 0.8, ...] # [地位低, 女性特征低, 男性特征高...]
vector_woman  = [0.1, 0.9, 0.1, ...] # [地位低, 女性特征高, 男性特征低...]

# 向量计算
result = vector_king - vector_man + vector_woman
# 结果会在多维空间中落在一个特定坐标上
# 然后我们去词汇库里找离这个坐标最近的词，发现正是 "Queen"！
closest_word = find_nearest_vector(result) # 返回 "Queen"
```
> **⚠️ 勘误提示**：原文说“不是词向量完全一致，而是最接近皇后的词向量”，**这句话非常严谨！** 运算结果确实不会与 Queen 的原始向量 100% 重合，而是空间距离最近。

## 四、 距离度量：如何计算两个词有多像？

> **💡 原文回放**：用余弦函数进行计算...首先点积，再把2个词向量开平方，最后用点积除以开平方，得到余弦值。
> **⚠️ 勘误提示**：这里的数学描述有些模糊和不准确，容易引起误解。

**正确的余弦相似度（Cosine Similarity）公式**：
不是简单地“把词向量开平方”，而是计算两个向量的**点积（Dot Product）**，再除以它们各自**模长（L2 Norm，即向量的长度）的乘积**。

```python
# 工程师视角的 Python 实现
import numpy as np

def cosine_similarity(vec_a, vec_b):
    # 1. 计算点积 (Dot Product): 对应位置相乘再求和
    dot_product = np.dot(vec_a, vec_b)
    
    # 2. 计算模长 (L2 Norm): 各个元素的平方和，再开根号
    norm_a = np.linalg.norm(vec_a) 
    norm_b = np.linalg.norm(vec_b)
    
    # 3. 计算余弦值
    return dot_product / (norm_a * norm_b)
```
**结果解读**：
- **1**：方向完全相同（极度相似）
- **0**：正交（毫无关系）
- **-1**：方向完全相反（反义词）

## 五、 从正则匹配到语义检索（Semantic Search）

> **💡 原文回放**：这就让我想到了搜索引擎的搜索，以前写项目的时候，都会用模糊匹配，关键词匹配。比如，搜索苹果的时候，除了手机也会有水果...但是有了词向量，ai可以根据词向量来判断两个单词是否相近，从而返回更准确的结果。
> **🔍 扩展应用**：这是一个极佳的工程联想！

在传统的软件开发中，我们做搜索通常依赖 **关系型数据库（如 SQL 的 `LIKE '%苹果%'`）** 或是 **全文检索引擎（如 ElasticSearch 基于倒排索引和 BM25 算法）**。这些属于**字面量匹配（Lexical Search）**。
- **痛点**：搜“手机”，出不来“iPhone”，因为字符串不匹配；搜“苹果”，既出“水果”又出“数码产品”，因为缺乏语境。

**现代的向量检索（Vector Search / Semantic Search）**：
有了 Embedding（词向量）技术后，我们可以把一整句话也变成一个长向量。
1. **语义理解**：如果你搜“被咬了一口的科技公司标志”，传统搜索会懵，但 AI 会把这句话转换为一个向量。
2. **向量数据库**：我们在向量数据库（如 Milvus, Qdrant, Pinecone）中计算这个向量与所有文档向量的**余弦相似度**。
3. **精准召回**：系统会发现这句话的向量与“苹果公司（Apple Inc.）”的向量距离极近，从而准确返回结果。

> **💡 总结**：AI 并不“懂”人类的语言，它只是通过海量的阅读，将词汇映射到了一个高维的数学空间中。在这个空间里，语义的相近体现为几何距离的相近。这就是大语言模型（LLM）能听懂我们说话的基石。
